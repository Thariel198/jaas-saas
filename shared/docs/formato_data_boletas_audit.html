Owner: shared/data_boletas_repo.py  |  Ruta: shared/data_boletas_audit.xlsx  |  Lifecycle: permanente — nunca se borra ni se edita  |  Hojas: 1 (Audit)

Propósito

Registro inmutable de toda mutación a DATA_boletas.xlsx.
Una fila por cada llamada exitosa a repo.apply_correction(...). Sirve para responder en tiempo constante:
  • ¿Cuándo cambió el Convenio del predio A-12?
  • ¿Qué agentes/módulos han tocado el predio B-21 este mes?
  • ¿Cuál era el VALOR_ANTERIOR antes del último cambio en MES_ANTERIOR de C-14?
  • ¿Cuántas correcciones se aplicaron por origen 4b_reclamos vs futuros agentes?
Nunca se modifica una fila ya escrita. Si una corrección fue errónea, se aplica una nueva corrección — que queda como otra fila en este audit.
Quién puede escribir este archivo: únicamente shared/data_boletas_repo.py.
Ningún otro script abre este archivo en modo escritura. Si necesitas registrar una mutación, llamas repo.apply_correction(...) — el repo se encarga del append.

VALOR_ANTES / VALOR_DESPUES son los valores observados por el repo en el momento del write. El "antes" se lee con un read dentro de la misma operación atómica que hace el write — así antes es siempre consistente con lo que el siguiente caller verá si llama get_predio.

Convenciones por source

SOURCE AUDIT_REF — formato esperado Qué identifica
4b_reclamos MESA|FECHA_COBRO (ej. mesa_2|2026-05-05) El cobro de efectivo que disparó el reclamo (con MZ/LT ya en columnas propias)
agente_* UUID del evento que disparó la corrección El evento agentic que decidió aplicar
conciliador_* ID del lote de conciliación El proceso batch que detectó la discrepancia
El formato de AUDIT_REF es libre por source. El repo solo exige que sea no vacío. Cada source documenta su propia convención — la idea es que sea suficiente para volver a encontrar el evento original en su sistema de registro.

Invariantes del archivo

Invariante Garantía
Append-only Una vez escrita, una fila nunca se modifica ni se borra. Las "rectificaciones" son nuevas filas.
Atomicidad antes/después VALOR_ANTES y VALOR_DESPUES corresponden exactamente al estado de DATA_boletas inmediatamente antes/después del write del repo.
Source obligatorio Ninguna fila tiene SOURCE vacío. El repo rechaza llamadas sin source identificable.
Ordenado por timestamp Las filas se agregan al final en orden cronológico de write. No se reordena nunca.

Señales de alerta

Señal Interpretación
Una fila tiene VALOR_ANTES = VALOR_DESPUES Write no-op — el caller debería haber detectado idempotencia antes de llamar. No es un bug del repo, pero indica desperdicio.
SOURCE desconocido Algún módulo nuevo está llamando al repo sin estar registrado en convenciones. Documentar antes de aceptar más writes desde ahí.
AUDIT_REF vacío El repo debería rechazar esta llamada — si aparece, hay un bug en validación de inputs.
Mismo (MZ, LT, CAMPO) con writes consecutivos en minutos Posible bucle de corrección o agente mal configurado. Revisar AUDIT_REF para entender el origen.
Crecimiento descontrolado (>10k filas/mes) Considerar particionar por año o migrar a base de datos. El Excel deja de ser práctico arriba de ~100k filas.
