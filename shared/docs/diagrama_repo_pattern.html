shared/data_boletas_repo — patrón repository

Único escritor de DATA_boletas.xlsx · audit centralizado · API estable para todos los módulos

Por qué existe el repo: en agentic SaaS, múltiples agentes/módulos van a leer y mutar DATA_boletas (resolución de reclamos hoy; auto-resolvers, importadores, conciliadores mañana). Si cada uno escribe directo, la disciplina (backup, log, validación) no escala y el audit trail se fragmenta. El repo encapsula la mutación: una sola puerta, audit obligatorio, lista para migrar a Postgres/servicio sin tocar consumidores.

shared/data_boletas_repo.py
API pública · internals encapsulados
API pública
read_padron() → DataFrame
Retorna el padrón completo. Sin side-effects.
get_predio(mz, lt) → dict | None
Retorna la fila del predio o None si no existe.
apply_correction(mz, lt, campo, valor, *, source, audit_ref, motivo) → AppliedDelta
Mutación atómica: lee valor previo, hace backup, escribe nuevo valor, registra audit. Retorna {valor_antes, valor_despues, ts, audit_id}. Falla si predio o campo no existen.
Internals (lo que el caller no ve)
  • Validación: el campo existe en DATA_boletas; el predio (mz, lt) existe.
  • Backup automático antes de cualquier write — 3_boletas/backup/DATA_boletas/...
  • Append al audit log — shared/data_boletas_audit.xlsx
  • Lock por archivo (futuro: lock por predio + transacciones).
  • Idempotencia: aplicar el mismo audit_ref dos veces produce un solo write efectivo.

Audit obligatorio No hay forma de mutar sin pasar por el audit log. Cada cambio queda con timestamp, source, audit_ref y motivo trazables.
Lecturas también vía repo resolucion.py y validacion_*.py ya leen vía repo. Es un acuerdo de cohesión, no de seguridad — preserva la abstracción para futuras migraciones.
Listo para escalar Cuando DATA_boletas migre a Postgres o microservicio, solo el repo cambia. Callers siguen llamando la misma API.
