Invariantes del módulo

Preservación manual Clave (MESA, MZ, LT, FECHA_COBRO). Nunca sobreescribir lo que el supervisor llenó en re-corridas.
Idempotencia Re-correr sin cambios en inputs no modifica ningún output. El repo además garantiza idempotencia de aplicación: aplicar el mismo audit_ref dos veces produce un solo write efectivo.
Una historia por reclamo Todo el ciclo vive en una sola fila de trazabilidad_reclamos.xlsx. Sin JOINs para responder "¿qué pasó?".
Corrección null para RECHAZADO CAMPO / VALOR_ANTERIOR / VALOR_APLICADO quedan vacíos si ESTADO_FINAL = RECHAZADO. No se llama al repo en estos casos.
DATA_boletas solo vía repo Ningún script de este módulo abre DATA_boletas.xlsx directamente — ni para leer ni para escribir. Todo pasa por shared/data_boletas_repo. Sustituye la antigua invariante "no modifica orígenes".
Audit obligatorio en cada write Cuando aplicar_correcciones.py llama al repo, este registra source=4b_reclamos, audit_ref=(MESA|FECHA_COBRO), motivo=RESOLUCION en shared/data_boletas_audit.xlsx.
Verificación no bloquea validacion_resolucion_correcciones.py reporta discrepancias pero nunca modifica nada. Es auditoría, no control de flujo.
