Contrato de formato — resolucion_reclamos_YYYY-MM.xlsx

Módulo: 4b_reclamos  |  Script: resolucion.py  |  Ruta: outputs/resolucion_reclamos_YYYY-MM.xlsx  |  Lifecycle: se regenera con cada corrida de resolucion.py  |  Hojas: 1 (Correcciones)

Propósito

Hoja de trabajo para que el supervisor decida la corrección.
resolucion.py auto-popula CAMPO y VALOR_ACTUAL cruzando reclamos_YYYY-MM.xlsx con DATA_boletas.xlsx.
El supervisor solo necesita llenar tres campos.

  • Sistema auto-llena: NOMBRE (de DATA_boletas), CAMPO (componente disputado) y VALOR_ACTUAL (lo que registra DATA_boletas hoy).
  • Supervisor llena: VALOR_A_CORREGIR, RESOLUCION, ESTADO, FECHA_RESOLUCION.
  • El cruce con DATA_boletas usa la clave (MZ, LT) — única en el padrón (575 predios, 0 duplicados).
  • Si el predio no se encuentra en DATA_boletas, NOMBRE / CAMPO / VALOR_ACTUAL quedan vacíos y se imprime una advertencia en el log.
  • Este archivo es el master state del cierre: el supervisor decide aquí ESTADO + FECHA_RESOLUCION. main.py sincroniza esos valores hacia reclamos_YYYY-MM.xlsx al correr, así no hay que tocar dos archivos.
  • Cuando main.py cierra el reclamo, toma CAMPO / VALOR_ACTUAL / VALOR_A_CORREGIR de este archivo (lookup por (MESA, MZ, LT, FECHA_COBRO)) y los registra en trazabilidad_reclamos.xlsx.
  • ★ Input del aplicador: aplicar_correcciones.py también lee este archivo — filtra las filas con ESTADO=RESUELTO + CAMPO no vacío + VALOR_A_CORREGIR no vacío, y llama repo.apply_correction(MZ, LT, CAMPO, VALOR_A_CORREGIR, motivo=RESOLUCION) por cada una.
★ Este archivo es input de dos consumidores:
   1. main.py (2da corrida) — para cerrar a trazabilidad
   2. aplicar_correcciones.py — para bajar VALOR_A_CORREGIR a DATA_boletas vía repo
Por eso es crítico que el supervisor llene RESUELTO + CAMPO + VALOR_A_CORREGIR + RESOLUCION para cada fila resuelta — el motivo (RESOLUCION) se propaga al audit log del repo.

Historia de las columnas (orden): identificas el predio → ves qué cobro generó el reclamo → confirmas el mes de origen → lees qué reclamó el usuario y cómo lo clasificaste → ves qué registra el sistema hoy → decides qué debería registrar y por qué → cierras.
NOMBRE: extraído de la columna NOMBRES de DATA_boletas.xlsx al cruzar por (MZ, LT).
Permite al supervisor confirmar que el predio es el correcto antes de aplicar la corrección.
CAMPO y VALOR_ACTUAL (cyan): auto-llenados por resolucion.py.
CAMPO = nombre del componente disputado (ej. Convenio, MES ANTERIOR).
VALOR_ACTUAL = valor exacto que registra DATA_boletas.xlsx al momento en que se corrió resolucion.py.
Si el predio no se encuentra en DATA_boletas, ambos quedan vacíos.
VALOR_A_CORREGIR, RESOLUCION, ESTADO, FECHA_RESOLUCION: los llena el supervisor manualmente solo en este archivo.
VALOR_A_CORREGIR = lo que debería registrar DATA_boletas (puede ser 0.00 si la deuda no existía).
RESOLUCION = explicación de la decisión (ej. "Verificado con recibo físico N°88").
ESTADO = dropdown PENDIENTE / EN_REVISION / RESUELTO / RECHAZADO. Esta es la decisión maestra del cierre — main.py la propaga a reclamos_YYYY-MM.xlsx.
Para RECHAZADO: VALOR_A_CORREGIR puede quedar vacío (no hubo corrección que aplicar).
