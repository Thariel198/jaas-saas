Columna ACCION — qué hizo el script con la fila del reporte:
INSERT — agregó el usuario a lista_sin_servicio.xlsx
UPDATE — promovió el TIPO existente (típicamente SIN_AGUA o EN_INVESTIGACIÓN → SIN_MEDIDOR)
SKIP_DUPLICATE — la decisión coincide con el estado actual · idempotente, no se duplica
NO_OP_ERROR_CAPTURA — no toca la lista · el supervisor agrega la lectura al template y re-corre main.py
NO_OP_IGNORAR — no toca la lista · queda solo el rastro forense en este audit
Columna ORIGEN_PROPUESTA — quién pre-llenó la decisión:
supervisor — DECISIÓN llenada manualmente por el supervisor en el reporte (caso normal)
auto — auto-promoción del sistema: 3 ciclos consecutivos sin lectura · prefijo auto: 3 ciclos sin lectura en NOTAS
default_sin_agua — pre-llenado automático cuando el usuario estaba como SIN_AGUA en la lista · prefijo default: SIN_AGUA → sin_servicio en NOTAS
Columna CICLO_ORIGEN:
Formato YYYY-MM · ciclo N — el mes y ciclo del reporte validacion_ausencias_YYYY-MM.xlsx donde nació la decisión.
Idempotencia y append-only:
Cada corrida de actualizar_lista.py agrega filas pero nunca elimina ni edita previas. El re-write con grupos se hace leyendo todas las filas existentes y reescribiendo el archivo completo — el resultado es indistinguible de un append puro, con la ventaja de migrar v1 → v2 sin pasos manuales.
