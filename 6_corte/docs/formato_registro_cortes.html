Propósito

Estado persistente de usuarios con servicio físicamente cortado.

  • seguimiento.py escribe automáticamente una fila con ESTADO=CORTADO cuando confirma el corte físico (idempotente por MZ+LT+MES_INICIO_CORTE).
  • El supervisor agrega manualmente ESTADO=EXONERADO para locales comunales o usuarios con enfermedad — quedan fuera del flujo indefinidamente o hasta que el supervisor los reactive.
  • El supervisor actualiza manualmente ESTADO=REACTIVADO + llena MES_REACTIVACION cuando el usuario paga toda la deuda acumulada + S/40, o cuando termina la exoneración.
  • generar_lista.py lee este archivo al inicio y excluye completamente a todos los usuarios con ESTADO=CORTADO o ESTADO=EXONERADO — no aparecen en lista_corte, no se les cobra penalidad.
  • No hay acuerdos de pago parciales — estados posibles: CORTADO, EXONERADO, REACTIVADO.
Distinción crítica: Un usuario CORTADO o EXONERADO NO aparece en lista_corte (ni siquiera con EJECUTAR_CORTE=NO). La exclusión es total. Diferente a EN_REVISION, que sí aparece con EJECUTAR_CORTE=NO.
Primer ciclo: Si el archivo no existe, generar_lista.py lo ignora sin error (0 exclusiones). El archivo se crea en el primer ciclo donde haya cortes físicos confirmados.

Idempotencia

Clave de idempotencia: (MZ, LT, MES_INICIO_CORTE)

Antes de insertar una fila, seguimiento.py verifica que no exista ya un registro con el mismo MZ + LT + MES_INICIO_CORTE. Si ya existe, no duplica. Re-correr el script el mismo día no altera el archivo.
